<implementation_scope>
    <owned_surfaces>
        - `package.json`: convert to a Bun workspace root while preserving
          compatibility scripts and dependency declarations.
        - `bun.lock`: update through `bun install` when package manifests
          change.
        - `tsconfig.json` and new package/app TypeScript configs: introduce a
          root base config and package-local configs as needed.
        - `src/`: source of current runtime files to migrate out of the old
          single-package tree.
        - `apps/cli/`: final home for the CLI binary, main dispatcher, command
          modules, usage, args, and CLI defaults.
        - `apps/dashboard/`: final home for the React/Vite dashboard frontend.
        - `apps/dashboard-server/`: final home for the Bun dashboard API,
          static serving, process-control, dashboard data projection, and
          trusted-report integration.
        - `packages/core/`: final home for shared runtime core: `state`,
          `board`, `shell`, `report`, `objdiff`, `handoff`, `env`, and shared
          `types`.
        - `packages/agents/`: final home for director, worker, PR-review,
          knowledge-curator, runtime Pi invocation, templates, schemas, and
          agent context selection.
        - `packages/knowledge/`: final home for TypeScript knowledge graph,
          resource map, decomp context, graph ranking, and path helpers that
          operate over root `knowledge/`.
        - `packages/ui-contract/`: final home for shared dashboard API and UI
          payload types used by browser and server.
        - `tests/`: update smoke imports and add migration-specific coverage if
          needed.
        - `Makefile`: update commands to call root compatibility scripts or
          package-local scripts correctly.
        - `README.md` and `docs/20-implementation/**`: update source-layout,
          CLI, UI, knowledge, agents, and state references.
        - `.gitignore`: update ignored build outputs, package dist folders, and
          generated UI/server outputs as needed.
        - `objectives/monorepo-restructuring/artifacts/`: write baseline,
          migration map, validation, and handoff artifacts.
    </owned_surfaces>

    <read_only_references>
        - `AGENTS.md`: repo instruction source; read and obey, do not rewrite
          unless the user asks.
        - `docs/10-system-design/**`: read to preserve system behavior; only
          update if implementation references become stale.
        - `knowledge/sources/**/data/` and `knowledge/resource_graph/graph.sqlite`:
          runtime knowledge data should not be churned for a layout-only
          migration.
        - `.decomp-orchestrator-state/`, `.pi-sessions/`, `.pi-agent/`,
          `local.env`, and `global.env`: local runtime/private state.
        - Parent Melee checkout: the objective restructures this orchestrator
          repo, not upstream Melee source.
        - User-modified files unrelated to the migration: do not revert or
          overwrite.
    </read_only_references>

    <generated_outputs>
        - `objectives/monorepo-restructuring/artifacts/baseline_summary.json`:
          pre-migration commands, pass/fail state, known dirty worktree notes,
          and current script/import assumptions.
        - `objectives/monorepo-restructuring/artifacts/migration_map.json`:
          old path, new path, package owner, import rewrite policy, and phase
          for every moved source area.
        - `objectives/monorepo-restructuring/artifacts/dependency_graph.json`:
          workspace package dependencies and any detected forbidden edges.
        - `objectives/monorepo-restructuring/artifacts/validation_summary.json`:
          final validation commands, outputs, timestamps, and residual risks.
        - `apps/dashboard/dist/` or the selected dashboard build output:
          generated by `bun run ui:build`; ignored unless a local convention
          requires otherwise.
        - Package `dist/` directories if introduced: generated build outputs,
          ignored unless publish artifacts are deliberately tracked.
    </generated_outputs>

    <commands_and_entrypoints>
        - `bun install`: refresh workspace dependency links and lockfile after
          package manifests change.
        - `bun run check`: root all-TypeScript validation. Must pass at every
          broad phase gate.
        - `bun run smoke`: root vertical smoke. Must pass before completion and
          after phases that move CLI/core/agents/state imports.
        - `bun run ui:check`: dashboard frontend typecheck. Must pass after
          dashboard moves.
        - `bun run ui:build`: dashboard frontend build. Must pass after
          dashboard moves and before final server validation.
        - `bun run orch -- --help`: CLI entry smoke from the root
          compatibility command.
        - `bun run orch -- --repo-root testdata/smoke_repo --state-dir <tmp> status`:
          CLI/state path smoke against fixture data.
        - `bun run kg:sources`: knowledge command smoke after CLI/package path
          rewrites.
        - `bun run kg:status`: knowledge graph path smoke when the tracked
          graph database is available.
        - `make status`, `make check`, and `make smoke`: operator wrapper
          smoke after root script changes.
    </commands_and_entrypoints>

    <adjacent_surfaces_requiring_caution>
        - `src/ui/server.ts`: large file with many responsibilities; split only
          when it improves the move and keep endpoint behavior covered.
        - `src/knowledge/resources.ts` and `src/agents/context.ts`: current
          dependency cycle. Refactor the data flow before enforcing package
          boundaries.
        - `src/knowledge/paths.ts`: root/path assumptions must continue to
          resolve the orchestrator repo root, parent checkout, and root
          `knowledge/` consistently after files move.
        - `tests/smoke.ts`: broad smoke imports many internals; update it as a
          consumer of package exports, not private moved paths.
        - `Makefile`: command quoting and default variables are operator
          ergonomics; keep behavior stable.
        - `docs/` and `README.md`: update references after paths settle, not
          before the target map is implemented.
    </adjacent_surfaces_requiring_caution>

    <out_of_scope>
        - Scheduler strategy, worker prompt policy, scoring policy, and PR
          split semantics are behavior work, not layout work.
        - Dashboard visual redesign is out of scope except for fixing broken
          imports/build paths caused by the move.
        - Knowledge source corpus refresh, PR corpus refresh, and live tool
          runner expansion are out of scope unless a path migration breaks an
          existing command.
        - Publishing packages to npm or another registry is out of scope; all
          packages remain private workspace packages.
        - Creating a separate MCP server or long-lived API service is out of
          scope.
        - Migrating the orchestrator into the parent Melee repo is out of
          scope; this remains its own repository.
    </out_of_scope>

    <target_tree>
        - See `objectives/monorepo-restructuring/examples/target_tree.md` for
          the intended final file tree and dependency direction.
    </target_tree>
</implementation_scope>
